# 협력하는 객체들의 공동체
> 느낀점

1. 너무 실세계와 밀접한 용어로 설명을 하다보니 살짝 괴리감이 느껴진다
    - 객체와 클래스의 차이는?
    - 협력적인 객체라는 것은?
    - 객체가 자기자신을 책임진다는 것은?
    - 객체가 행동을 수행하는 방법(혹은 협력에 참여하는 방법)을 스스로 결정한다는 것은?
    - 객체의 개방성과 자율성이 명확하게 인지가 되지 않는다
        - 단어가 추상적이고 비슷한 개념이 미묘하게 다른 단어로 사용되어 어떤 개념인지 잘 안잡힌다

2. 커피공화국의 아침에서 협력 개념을 설명할 때 동의하지 않는 부분이 있었다
    - **커피 주문이라는 협력**에 참여한 사람들은 자신들의 역할과 책임을 다하고 있다
    - **요청과 응답**으로 구성됨
    - 협력의 성공은 특정한 역할을 맡은 각 **개인이 얼마나 요청을 성실히 이행**하는 가에 달려있다
        - 동의하지 않는다. 역할은 상황에 따라 유동적으로 변한다. 
        - 맡은바 수행을 성실히 하는가의 문제보다는 얼마나 기민하게 반응하고 대처하는지 대응능력이 더 중요
> 새롭게 배운 개념
- 객체지향이란 **시스템**을 객체로 분할하여 바라본다는 것
- 각 객체는 좋은 협력을 위해 **좋은 품질의 객체**여야 한다
- **유지보수가 쉽고 재사용이 용이**한 것이 품질의 지표
    - 객체는 충분히 개방적이고 자율적이어야 
> 기억하고 싶은 책 내용
## 이 책에서 깨고 싶은 편견들
### 1. **클래스 vs 객체**
- 사고의 전환이 필요
    - '코드를 담는 클래스'의 관점 -> '**메시지를 주고받는 객체**'의 관점
    - 어떤 클래스가 필요하냐 -> **어떤 객체들이 어떤 메시지**를 주고받는가
    - 클래스 
    - '**객체들의 협력관계**'를 '코드로 옮기는 도구'에 불과
    - '클래스의 구조'와 '메서드' -> '**객체의 역할, 책임, 협력**'에 집중하
    - 객체지향은 **객체를 지향**하는 것이지 클래스를 지향하는 것이 아니다
    - 객체지향의 중심은 클래스나 상속이 아니다!

## 이 책에서 말하고 싶은 것

### 1. **객체 지향 : '특정 역할'을 지닌 '자율적인 객체'들의 '협력'** 

1. 시스템
    - 상호작용하는 자율적인 객체들의 공동체
    - 객체를 이용해 시스템을 분할
2. 역할
    - 역할의 특징
        1. 여러 객체가 동일한 역할을 수행할 수 
        2. 역할은 대체 가능하다
        3. 한 객체가 여러 역할을 수행할 수 
    - 역할의 중요성
        - 역할은 유지보수가 쉽고 재사용한 협력관계를 구축하기위해 필요한 요소

3. 자율적인 객체
    - 자율적인 객체의 조건
        1. (행동을 위해 필요한) **상태**를 지님
        2. **스스로 자기 자신을 책임**지는 객체
            - 각 객체들은 행동을 수행하는 방법을 스스로 결정
            - 메시지를 수신한 객체는 메시지를 처리하는 메서드를 런타임 시점에 결정(이게 왜 중요하지?)
    - 자율적인 객체의 장점
        - 자율적인 객체로 구성된 공동체는 **유지보수가 쉽고 재사용이 용이**

4. 협력
    - 협력의 품질은 객체의 품질
        - 각 객체는 협력 내에서 정해진 역할을 수행
        - 객체는 충분히 개방적인 동시에
            - 다른 객체와 조화롭게 협력할 수 있어야
        - 충분히 자율적이어야
            - 협력에 참여하는 방법을 스스로 결정
    - 협력 수단
        - 메시지

### 2. **시너지! 전체는 부분의 합보다 크다**

- 객체지향의 다양한 측면들
    1. 상태와 행위를 캡슐화하는 **객체의 자율성**
    2. 메시지를 주고받으며 **공동의 목표**를 달성하기 위해 협력
    3. **연결완전성**



예시 : 커피공화국의 아침 - 역할, 책임, 협력

# 2. 이상한 나라의 객체
> 느낀점
> 새롭게 배운 개념
> 기억하고 싶은 책 내용

## **객체지향 패러다임**
- 목적 
    1. 실세계 문제에 내재된 **복잡성을 관리**
    2. 지식과 행동을 **구조화**
- How to
    - **객체**를 발견하고 창조
    1. 지식을 추상화 
    2. 추상화한 지식을 **객체** 안에 캡슐화 

---

## **객체의 특징(feat.앨리스)**
1. 앨리스는 **상태**를 가지며 상태는 변경가능하다
2. 앨리스의 상태를 변경시키는 것은 앨리스의 **행동**이다
    - 행동의 결과는 상태에 의존적이며 상태를 이용해 서술할 수 있다
    - 행동의 순서가 결과에 영향을 미친다
3. 앨리스는 어떤 상태에 있더라도 유일하게 **식별 가능**하다

<br>

### **상태와 프로퍼티**
1. **상태**로 '**행동의 과정과 결과**'를 단순하게 기술이 가능해짐
    - 과거에 발생한 행동의 이력을 통해 현재 발생한 행동의 결과를 판단하는 방법은 복잡하고 번거롭다. 
        - ex. 비행기 탑승 가능여부
            - 발권을 한 이력이 있는가? -> 항공권의 발권상태가 y인가?
        - ex. 자판기의 총 매출액
            - 자판기를 사용한 이력 -> 현재까지 투입된 금액의 상태
    - 상태를 이용하면 과거의 모든 행동 이력을 설명하지 않고도 행동의 결과를 쉽게 예측하고 설명할 수 있다

<br>

2. '**상태**'는 특정 시점에 '객체'가 가지고 있는 '**프로퍼티(정보)의 집합**' 
    - 객체의 구성
        1. 정적 프로퍼티와 동적 프로퍼티
            - 프로퍼티
                - 변경되지 않고 고정된 것, 정적
            - 프로퍼티의 값
                - 시간이 흐름에 따라 변경, 동적
        2. '단순한 값'과 '다른 객체를 참조하는 링크' 
            1. 속성(단순한 값)
                - ex. 숫자, 문자열, 양, 속도, 시간, 날짜, 참/거짓
                - ex. 앨리스의 **키와 위치**
                - 그 자체로 독립적인 의미를 가지기보다는 다른 **객체의 상태를 표현**하는데 사용됨
            2. 객체
                - 객체를 사용해 다른 객체의 상태를 표현
                - ex. 앨리스가 현재 **음료**를 들고 있는가?


### **행동**

1. 행동의 목적
    - 외부의 요청에 응답
    - 다른 객체와 협력에 참여
        - 행동은 외부에 가시적이어야 한다
2. 행동의 결과
    - 객체는 자신의 상태를 변경 
    - 다른 객체에게 메시지를 전달

<br>

3. '**상태**'와 '**행동**'은 서로에게 영향을 미친다
    - 객체의 행동은 상태를 변경시킨다(부수효과가 생김)
        - ex. 앨리스가 케익을 먹는다 -> 키가 작아짐
        - ex. 앨리스가 문을 통과 -> 앨리스의 위치가 변화
    - 객체의 행동은 상태에 영향을 받는다
        - 객체의 행동의 결과는 객체의 상태에 의존적이다
        - ex. 음료를 마신 후의 앨리스의 키는 음료를 마시기 전의 앨리스의 키보다 작아져야 함

<br>

4. 객체는 협력하는 과정에서 **다른 객체의 상태**도 변경할 수 있다
    - ex. 앨리스가 음료를 마시면 앨리스 키가 작아지는 동시에 음료의 양도 준다.
    1. 객체 자신의 **상태 변경**
    2. 행동 내에서 협력하는 **다른 객체에 대한 메시지 전송**


### **'캡슐화'와 자율성, 협력**
1. **상태를 캡슐화**하는 것은 **객체의 자율성**을 높이고 **협력을 단순하고 유연**하게 만든다
    - ex. 메시지 송신자는 메시지 수신자의 상태 변경에 대해 알지 못한다
        - 앨리스 객체 ---(drunken(quatity))---> 음료 객체
        - 음료에게 메시지를 전송하는 앨리스 객체는 음료의 상태를 알지 못한다
    1. 캡슐화
        - 객체는 상태를 캡슐 안에 감춰 외부로 노출하지 않는다
            - 객체가 외부에 노출하는 것은 행동 뿐
            - 외부에서 객체에 접근할 수 있는 유일한 방법도 행동
    2. 상태를 캡슐화하는 것은 객체의 자율성을 높인다
        - 자율적인 객체는 스스로 판단하고 스스로 결정하기 때문에 객체의 지능도 높아진다
    3. 협력에 참여하는 객체들의 지능이 높아질수록 협력은 유연하고 간결해진다

### **'식별자'와 값, 객체**

1. 식별자 : 객체를 다른 객체와 구분하는데 사용하는 객체의 프로퍼티
    - 값
        - 식별자를 가지지 않음
        - 두 인스턴스를 비교 : **상태**를 이용한 동등성 검사
    - 객체
        - 상태가 변경될 수 있음
        - 두 인스턴스를 비교 : **식별자**를 이용한 동일성 검사

<br>

2. 값과 객체의 차이점
    - 두 개념 모두 클래스를 이용해 구현되기 때문에 대부분 이 둘의 차이를 헷갈려한다
    - 객체지향 언어의 관점에서 값과 객체 모두 클래스로부터 생성된 객체이기 때문에 문맥에 따라 그 의미가 혼란스러울 수 있다.
        - 숫자는 Integer클래스로부터 생성된 객체이고 (값 객체)
        - 사람은 Person클래스로부터 생성된 객체다 (엔티티, 참조객체)
---
## **객체는 다른 객체와 협력하기 위해 존재한다**
- 객체의 행동은 협력에 참여하는 유일한 방법이다.
    1. 객체지향 설계는 **어플리케이션에 필요한 협력**을 생각하고 
    2. **협력에 참여하는데 필요한 행동**을 생각한 후
    3. **행동을 수행할 객체**를 선택
    
## **행동을 상태보다 먼저 고민하라**

1. 객체지향에 갓 입문한 사람들이 쉽게 빠지는 함정은 상태를 중심으로 객체를 바라보는 것이다
    - ex.앨리스 객체에 필요한 상태를 찾고 **키와 위치**를 앨리스에 추가한다
        - 그리고 키와 위치를 **변경하거나 조회할 수 있는 행동**이 무엇인지 고민한다
    - 상태를 먼저 결정하고 행동을 나중에 결정하는 방법은 설계에 나쁜 영향을 준다

<br>

2. 상태를 결정하고 행동을 나중에 결정하는 설계의 문제점
    1. 캡슐화가 저해됨
        - 상태가 객체 내부로 깔끔하게 캡슐화되지 못하고 공용 인터페이스에 노출되버릴 확률이 높아진다
    2. 객체를 협력자가 아닌 고립된 섬으로 만든다
        - 객체가 필요한 이유는 앱 문맥 내에서 다른 객체와 협력하기위해서
        - 상태를 먼저 고려하면 문맥에서 벗어난채 객체를 설계함으로써 협력에 적합하지 못한 객체를 창조하게 된다
    3. 객체의 재사용성이 저해
        - 객체의 재사용성은 다양한 협력에 참여하는 능력에서 나온다


- **책임주도 설계**
    - 협력 안에서 객체의 행동 : 객체가 협력 안에서 완수해야 하는 책임
    - 결국, 어떤 책임이 필요한가? 결정하는 과정이 전체 설계를 주도해야.

## **객체지향 설계는 의인화를 통해 현실과 전혀 다른 새로운 세계를 창조한다**
- 객체지향이란 현실 세계의 모방이라고 보는 관점(**추상화**)은 한계가 있다
- 현실 속 객체와 소프트웨어 객체 사이의 가장 큰 차이점은 **의인화**
    - 현실의 수동적 존재가 소프트웨어 객체로 구현될 때는 능동적으로 변한다는 것
