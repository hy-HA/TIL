# 1. 프로그래밍 언어 & 운영체제

> 느낀점
- 개발자는 기획자가 의도한 대로 어플과 웹이 작동되도록 프로그래밍 언어를 통해 운영체제 위에 돌아가는 프로그램을 만들고 관리하는 사람들
- 회사의 방향과 지점의 성장을 조율하면서 회사의 매출 향상에 도움이 되도록 지점을 관리하는 관리자와 크게 다르지 않다.

- 고등학교 때부터 배달로 돈을 벌어온 그들의 언어를 배우고, 성향을 이해하고, 니즈를 파악하고, 업계의 생태를 파악하고.
- 지금은 컴퓨터의 언어를 배우고, 생태계를 이해하는 것. 크게 다르지 않다. 

> 새롭게 배운 개념
- IDE : 개발을 하기 위한 모든것들을 제공해주는 환경. 
    - 안드로이드 스튜디오, xcode, 이클립스, 파이참 등,,
- CPU & 메모리(RAM) & 보조기억장치 (HDD(하드디스크),SDD)
    - CPU
        - 컴퓨터의 머리
    - 메모리(RAM)
        - CPU의 개인작업공간
        - 작업이 필요한 데이터 뭉치를 보조기억장치에서 메모리로 한번 옮겨놓고 메모리 안에서 작업
    - 보조기억장치 (HDD(하드디스크),SDD)
        - 컴퓨터의 창고
        - 전원이 꺼져도 데이터는 이곳에 남아있음
        - 이 부품들은 메인보드에 끼워져있음
- 운영체제
    - 운영체제는 사람 대신 하드웨어를 관리해줌
    - 하드웨어의 용량을 확인할 수 있는 것도 운영체제가 보조기억장치를 관리하고 있기 때문
    - 운영체제가 하드웨어를 컨트롤하고, CPU와 메모리를 관리해서 사람은 클릭 몇번으로 파워포인트를 실행하고 카톡을 설치

# 2. 네트워크, 클라이언트, 서버
> 느낀점

- 네트워크의 발전으로 세상의 트렌드를 모두가 공유하고 이야기할 수 있는 배경이 만들어지는 시대에 태어난 것이 참 다행이면서 재미있다. 나는 이런 시대에서 과연 뭘 할 수 있을까. 
- 리눅스 운영체제를 무료로 배포한 리눅스 토발즈라는 사람에 대해 더 알고 싶다. 의도는 무엇이었을까? 그 결정은 인터넷의 발전에 어떤 영향을 끼쳤을까?
> 새롭게 배운 개념
- 리눅스 운영체제를 사람들이 발전시키면서 여러 버전이 나옴
    - 우분투
    - 레드햇 : 리눅스를 개량해서 유료로 판매하는 회사
    - 안드로이드 : 구글이 리눅스를 모바일 운영체제 형태로 개량해서 발전시킨 운영체제

> 기억하고 싶은 책 내용

## 1. 국민학교와 초등학교 시절 이야기(LAN -> MAN -> WAN -> 3G)
- 컴퓨터를 연결하면 할 수 있는 일이 많아진다
- 게임산업은 네트워크의 발전에 크게 기여했음
## 2. 한강에서 카톡을 다운받고 실행하면 일어나는 일
- 사용자가 앱스토어에 카톡 다운 요청
- 다운 요청이 wan을 따라 가까운 기지국으로 요청
- 앱스토어(애플)는 애플 서버로 요청이 전달됨
- 애플 서버에서 카톡파일을 기지국에 전달, 사용자의 핸드폰에 다운 완료
<br>
- 컴퓨터의 보조 기억장치에 카톡 실행파일이 저장됨
- 사용자가 카톡 아이콘을 클릭, 실행에 필요한 부분이 메모리로 옮겨짐
- cpu가 그 데이터들을 처리하여 카카오톡이 동작

## 3. 당신은 뭘 계속 달라고 하고, 누군가는 계속 뭘 준다
- 클라이언트의 요청이 몰리게 되면 서버는 다운되기 쉽다
    - CPU가 처리할 수 있는 한계를 넘어서는 경우
    - 메모리의 공간을 모두 쓰게 되는 경우
- 따라서 서버를 여러 군데 분산시켜 두어 수많은 클라이언트가 24시간 365일 끊김없이 사용할 수 있도록 함

## 4. 우분투
- 운영체제 : 리눅스, 맥os, 윈도우
- 운영체제는 cpu,메모리,보조기억장치들을 사용자가 신경쓰지 않도록 관리함
- 운영체제 위에서 프로그램들이 돌아감
- 서버는 리눅스라는 운영체제 위에서 돌아감. 
    - 리눅스는 무료이기 때문
    - 리누스 토발스는 리눅스와 깃을 만듦
- 리눅스 운영체제를 사람들이 발전시키면서 여러 버전이 나옴
    - 우분투
    - 레드햇 : 리눅스를 개량해서 유료로 판매하는 회사
    - 안드로이드 : 구글이 리눅스를 모바일 운영체제 형태로 개량해서 발전시킨 운영체제
- 윈도우와 맥 os가 리눅스보다 비싼 이유
    - 바탕화면이라는 개념과 폴더와 파일의 개념이 만들어짐
    - 마우스 개념도 더해짐
    - 명령어를 몰라도 폴더안에 파일을 넣고 휴지통에 버리는 직관적이고 쉬운 방법으로 컴퓨터를 활용할 수 있게됨

# 3. API와 JSON

> 느낀점

- 같은 회사 사람이어도 서버 개발자와 클라이언트 개발자는 API문서를 통해 소통하는구나. 
- 면대면 미팅이 익숙한 나에게 가장 취약한게 메일, 문서를 통한 소통이다. 
- 문서로는 쉽게 놓칠 수 있는 부분을 짚어주는 기능이 없으니까. 
- 이 부분은 어떻게 보완할 수 있나?

> 새롭게 배운 개념

- API 
    - 클라이언트, 서버 등 서로 다른 프로그램에서 요청과 응답을 주고 받을 수 있게 만든 체계
    - 소프트웨어가 다른 소프트웨어를 쓰기 위해 중간에 필요한 체계
    - 기능을 사용하기 위해 주소로 요청을 보내면 응답을 해주는 소프트웨어끼리의 체계

- RESTful API
    - 이전보다 주소 개수가 줄어듬
    - CRUD를 하나의 주소로 관리
        - 요청을 보낼 때 어떤 요청을 보냈는지 알 수 있는 스티커(메소드)를 붙여서 보냄
    - 모든 회사에서 통용되는 절대규칙이 아닌 일종의 사회 운동으로, 상황마다 다양한 방식으로 변형해서 사용

- API 문서
    - 서버 개발자와 클라이언트 개발자가 협업하고 있는 회사라면 API문서가 존재
    - API문서를 통해 협업중인 프로젝트가 어떻게 진행되고 있는지 파악 가능

> 기억하고 싶은 책 내용

## 1. POST와 GET

- API 
    - 클라이언트, 서버 등 서로 다른 프로그램에서 요청과 응답을 주고 받을 수 있게 만든 체계
    - 소프트웨어가 다른 소프트웨어를 쓰기 위해 중간에 필요한 체계
    - 기능을 사용하기 위해 주소로 요청을 보내면 응답을 해주는 소프트웨어끼리의 체계

- 서버 개발자는 서버 프로그램을 통해 서버 주소별 수행 기능을 정의
    - API를 만들 때 데이터를 주고 받는 기능도 설정
    - API를 통해 요청과 응답을 주고 받으 ㄹ때 데이터도 같이 담김

- CRUD별 주소
    - 컴퓨터주소/create
    - 컴퓨터주소/read
    - 컴퓨터주소/update
    - 컴퓨터주소/delete

- RESTful API
    - 이전보다 주소 개수가 줄어듬
    - CRUD를 하나의 주소로 관리
        - 요청을 보낼 때 어떤 요청을 보냈는지 알 수 있는 스티커(메소드)를 붙여서 보냄
    - 모든 회사에서 통용되는 절대규칙이 아닌 일종의 사회 운동으로, 상황마다 다양한 방식으로 변형해서 사용

- CRUD별 메소드
    - create : POST
    - read : GET
    - update : PUT(전체)/PATCH(일부)
    - delete : DELETE

- HTTP 상태코드
    - 200번대 : 성공
    - 400번대 : 클라이언트의 요청에 문제가 있는 경우
    - 500번대 : 서버에 문제가 있는 경우

- SDK(Software Development Kit)
    - API를 제공해주는 다른 소프트웨어

## 2. 요청과 응답을 주고받을 때의 형식

- JSON
    - 클라이언트와 서버는 요청과 응답을 주고받고, 그때 필요한 데이터를 JSON형식으로 주고받음

## 3. API 문서

- 서버 개발자와 클라이언트 개발자가 협업하고 있는 회사라면 API문서가 존재
    - API문서를 통해 협업중인 프로젝트가 어떻게 진행되고 있는지 파악 가능
- open API
    - 기업에서 제공하는 API문서를 타 사에서도 사용할 수 있도록 오픈한 것

# 5. 웹

> 느낀점

> 새롭게 배운 개념

- HTML 
    - 서로 다른 운영체제나 애플리케이션을 사용 => 파일을 주고받을때 호환되지 않는 문제
    - 운영체제나 프로그램에 상관없이 일정한 형식이 언제나 동일하게 보이도록 하는 새로운 개념이 필요
    - 일정한 형식(HTML)으로 작성한 문서를 제안
    - HTML문서는 운영체제에 상관없이 브라우저만 있으면 웹사이트에 접속하여 동일한 정보를 볼 수 있음
    - 이를 통해 팀 버너스리는 모든 정보가 자유롭게 공유되는 세상, 즉 웹을 통해 누구나 쉽게 정보에 접근할 수 있는 위키백과 같은 모습을 꿈꾸었음
    ```
    주의할점
        - HTML은 프로그래밍 언어가 아님
        - 컴퓨터에게 특정 일을 시킬 수 있는 언어가 아닌
        - 브라우저가 볼 수 있는 문서를 적는 언어
    ```

- 하이브리드 애플리케이션
    - 각 회사들은 운영체제 안에 브라우저를 내장시켜 출시하기 시작
    - 애플리케이션 특정 부분에 브라우저를 올리는 방식으로 개발이 가능해짐(하이브리드 애플리케이션)
        - 애플리케이션 특정 부분에 HTML파일을 불러올 URL을 설정
        - 브라우저가 뜨고 그 브라우저는 HTML과 HTML에 연결된 파일들(HTML,css,자바스크립트)을 불러와서 보여줌

> 기억하고 싶은 책 내용

## 1. 웹은 어떻게 이루어져있고, 어떻게 동작할까

- 웹의 4가지 이슈
    - HTML, CSS, Javascript
    - 브라우저
    - 반응형 웹
    - 하이브리드 애플리케이션

- HTML 
    - 서로 다른 운영체제나 애플리케이션을 사용 => 파일을 주고받을때 호환되지 않는 문제
    - 운영체제나 프로그램에 상관없이 일정한 형식이 언제나 동일하게 보이도록 하는 새로운 개념이 필요
    - 일정한 형식(HTML)으로 작성한 문서를 제안
    - HTML문서는 운영체제에 상관없이 브라우저만 있으면 웹사이트에 접속하여 동일한 정보를 볼 수 있음
    - 이를 통해 팀 버너스리는 모든 정보가 자유롭게 공유되는 세상, 즉 웹을 통해 누구나 쉽게 정보에 접근할 수 있는 위키백과 같은 모습을 꿈꾸었음
    1. 태그
        - 정보를 표현하기 위한 코드
        - ```<h>``` ```<p>``` ```<ol>```등
    2. 주의할점
        - HTML은 프로그래밍 언어가 아님
        - 컴퓨터에게 특정 일을 시킬 수 있는 언어가 아닌
        - 브라우저가 볼 수 있는 문서를 적는 언어
    3. 문제
        - HTML이 정보전달에 초점을 맞추다보니 디자인 기능이 부족
        - 정렬을 맞추거나 도형을 만드는 디자인적 요소를 입히기 어려웠음
- CSS
    - HTML에 디자인을 입히는 코드
- 퍼블리싱 작업
    - HTML과 CSS를 합쳐 퍼블리싱 작업이라고 표현
- Javascript
    - 웹과 애플리케이션의 차이
        - 애플리케이션을 1.0.0에서 2.0.0으로 변경하려면 업데이트가 필요
        - 모바일이라면 심사도 필요
        - 웹은 서버의 원본을 바꾸면 됨
            - 새로고침하면 바뀐 HTML,CSS,Javascript,이미지 등이 다운로드 됨
    - 웹과 앱의 장단점
        - 웹
            - 장점 : 수정이 용이
            - 단점 : 네트워크의 영향을 크게 받음
        - 앱 
            - 장점 : 네트워크의 영향을 크게 받지 않음
            - 단점 : 수정이 용이하지 않음
                - 사람마다 애플리케이션의 버전이 다를 수 있음

## 2. 웹 개발하다가 못 해 먹겠다고 말하는 이유

- 브라우저
    - 브라우저는 설치가 필요함
    - 브라우저는 애플리케이션이라는 것
    - 애플리케이션은 사람마다 사용하는 버전이 다를 수 있음
    - 각 브라우저는 서로 다른 애플리케이션
        - 브라우저에 따라 그 안의 구현방식이 다름
        - HTML,CSS, 자바스크립트의 특정 기능이 버전별로, 브라우저별로 작동하지 않을 수 있음
        - 웹 프론트 개발자는 브라우저 버전과 종류에 맞춰 정상적으로 동작할 수 있도록 추가로 코드를 작성해야 함 (브라우저 버전의 파편화)

## 3. 반응형으로 코딩하면 더 비싸나요?

- 반응형 웹
    - 스마트 폰과 태블릿이 등장하면서 pc버전으로 웹을 볼때 화면이 잘려서 보이는 이슈가 발생
    - 오른쪽으로 넘기고, 확대/축소하며 불편하게 웹을 확인해야 했음
    - 그래서 m.naver.com / m.daum.net같이 주소 앞에 m이 붙는 모바일용 웹 페이지를 따로 만들었음
    - 하지만 모바일과 pc버전 웹 페이지를 따로 만들어서 출시하는 것은 불편. 
        - 디자인을 바꾸기 위해 서로 다른 css파일을 수정해야하기 때문. 
        - 작업이 중복되어 비효율적이고 버그가 생길수도 있음
    - 그래서 반응형 웹이 등장
        - 브라우저의 가로 넓이에 반응하여 구성요소가 변하는 기술

## 4. 애플리케이션 얘기를 하는데 왜 자꾸 웹 개발자에게 말하라는 거죠?

- 브라우저 
    - 5개의 유명한 브라우저 : 크롬, 익스플로러, 사파리, 오페라, 파이어폭스
    - 크롬 : 구글이 만든 브라우저
        - 안드로이드 : 구글이 만든 운영체제
    - 사파리 : 애플이 만든 브라우저
        - 맥os, ios : 애플이 만든 운영체제
    - 익스플로러 : 마이크로소프트가 만든 브라우저
        - 윈도우 : 마이크로소프트가 만든 운영체제

- 하이브리드 애플리케이션
    - 각 회사들은 운영체제 안에 브라우저를 내장시켜 출시하기 시작
    - 애플리케이션 특정 부분에 브라우저를 올리는 방식으로 개발이 가능해짐(하이브리드 애플리케이션)
        - 애플리케이션 특정 부분에 HTML파일을 불러올 URL을 설정
        - 브라우저가 뜨고 그 브라우저는 HTML과 HTML에 연결된 파일들(HTML,css,자바스크립트)을 불러와서 보여줌
    - 하이브리드 애플리케이션을 수정하고 싶은 경우
        1. 브라우저 위에서 돌아가는 부분
            - 서버에 있는 원본 HTML, CSS, 자바스크립트를 수정
            - 수정에는 용이
            - 하지만 네트워크에 종속되기 때문에 와이파이나 모바일 네트워크가 느린 공간에 가면 로딩시간이 길어져 사용자는 불편하다고 느낌.
        2. 네이티브 부분은 운영체제별로 다른 프로그램밍 언어를 통해 수정한 뒤 심사를 신청
            - ios : 스위프트, object-c
            - 안드로이드 : 자바, 코틀린
            - 네트워크를 최소한으로 이용하도록 코딩한다면 인터넷이 느린 환경에서도 빠르게 작동
            - ios심사를 거쳐야 한다는 큰산이 있고
            - 사용자 입장에서 직접 업데이트를 해주어야 함
